## 3. À DURCIR (sécurité & robustesse)

### Sécurité
- [app/api/webhooks/claap/route.ts:46](../app/api/webhooks/claap/route.ts#L46) — `received !== secret` vulnérable au timing attack → `crypto.timingSafeEqual()`
- [lib/cron-auth.ts:19](../lib/cron-auth.ts#L19) — même problème sur le Bearer cron
- **RLS Supabase absent** sur toutes les tables — toutes les requêtes passent par `SUPABASE_SERVICE_ROLE_KEY`. Aucune isolation tenant au niveau DB.
- **Webhook idempotence** : Netrows a `netrows_events_processed`, Claap n'a pas l'équivalent → race condition possible si retry avant l'upsert

### Fragilité
- **Pas de timeouts** sur les fetch externes ([lib/netrows.ts:48](../lib/netrows.ts#L48), [lib/hubspot.ts:4](../lib/hubspot.ts#L4), [lib/claap.ts:43](../lib/claap.ts#L43)) → wrapper `fetchWithTimeout()` partagé
- **JSON.parse non sécurisés** dans [lib/target-companies.ts:8,16,24](../lib/target-companies.ts) → Zod
- **Token refresh Gmail** OK mais sans logging si échec → user stuck silencieusement
- **`logUsage()` fire-and-forget** ([lib/log-usage.ts:15](../lib/log-usage.ts#L15)) → perte de données si DB down
- **Pas de retry/backoff** sur HubSpot (rate limit 100 req/10s) ni Anthropic

### Error handling
- Envelopes API incohérentes (`{ profiles, total }` vs `{ results, nextCursor }` vs flat) → standardiser `{ ok, data, error?, meta }` (rappel : le fetcher SWR swallow les non-2xx, donc l'API doit toujours renvoyer la shape attendue)
- 500 sur des erreurs upstream attendues (devrait être 502/503)
- `error.stack` jamais loggé côté serveur

## 5. UPGRADES (idées plus ambitieuses)

### Architecture
- **Migrations versionnées** avec timestamp prefix (actuellement ordre alphabétique ambigu)
- **PG cron + Edge Functions Supabase** pour échapper aux limites Netlify
- **Inngest / Trigger.dev / QStash** pour remplacer les Background Functions ad-hoc — observability + retry built-in
- **HubSpot webhooks** au lieu de polling LinkedIn weekly
- **pgvector** pour la recherche sémantique sur prospects/intel (au lieu de fuzzy-match maison)
- **Anthropic Batch API** pour scorer 1000 deals overnight à 50% du prix
- **Anthropic prompt caching** sur les guides (système) — gros gain $ et latence

### Frontend Next 16 / React 19
- **Server Components** pour deals, intel, prospecting (data fetching SSR + Suspense)
- **Server Actions** pour les mutations admin (guides, alerts, model preferences)
- **Parallel routes** (`@modal/(.)deal/[id]`) pour le panel deal et la modale HubSpot import
- **Streaming UI + `useTransition`** pour les long renders (kanban deals)
- **View Transitions** pour la nav stage→stage du kanban
- **Error boundaries** user-facing (actuellement 12 toasts pour 540 hooks → erreurs invisibles)

### Observability
- **Sentry / Better Stack** (zéro tracing actuellement, juste console.log)
- Erreurs critiques → channel Slack `#alerts`
- OpenTelemetry sur `/api/chat` et `/api/deals/score` (token usage, latency Claude, throughput)

### Validation
- **Zod** sur tous les inputs API et JSON.parse externes (HubSpot raw, Claap webhook payload, target-companies)

### DX
- **TanStack Query** plutôt que SWR — meilleur retry et granularité (mais migration coûteuse, à évaluer)
- **Tests** : aucun setup Vitest/Jest visible — au minimum, snapshot des prompts critiques (deal scoring, intel agents) pour détecter régressions
